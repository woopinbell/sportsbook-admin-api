# 002 · feat(api): 운영 액션 endpoint — /internal/v1 위임

## 개요

admin-api의 본체. 정산 void/replay·환불·한도 변경·마켓 close/reopen을 **각 도메인 service로 위임**한다. admin-api는 검증·헤더 전파·결과 relay만 하고 비즈니스는 안 한다(thin proxy). 시작 가설("그냥 호출하면 됨")이 깨진 지점이기도 하다 — 다운스트림 실제 계약이 문서와 달랐다(아래).

## 작업 순서

1. 다운스트림 컨트롤러를 **직접 읽어** 실제 경로/바디 확인.
2. `Uuid7` — 액션 id 생성기(헤더+Idempotency-Key+audit PK 공용).
3. `AdminContext`(actor + actionId + traceId) + 인자 resolver + `WebConfig` 등록.
4. service별 `RestClient` 4개(`DownstreamClientConfig`) + `AdminHeaders` 전파.
5. `*Client` 4개(settlement/wallet/risk/market) + `DownstreamCalls`(에러 변환).
6. 컨트롤러 4개 + DTO + `AdminExceptionHandler`(RFC 7807, 상태 relay).

## 작업 내역

### 다운스트림 계약이 문서와 달랐다 (설계 결정 3개)

코드를 열어보니 세 군데가 어긋나서 그 자리에서 결정:

| admin-api | 실제 다운스트림 | 결정 |
|---|---|---|
| 마켓 close | `/internal/v1/events/{eventId}/markets/{marketId}/close` | admin 경로에 **eventId 포함** |
| 환불 | wallet엔 `/credit`만 (`/refund` 없음) | 환불=`source=HOUSE_POOL` credit, actionId를 Idempotency-Key로 |
| suspend | user-service **부재** | endpoint 미노출 |

### HTTP 클라이언트는 RestClient

Spring 6.1 동기 클라이언트 `RestClient`(webflux 불필요). service당 1개 빈, base URL + 타임아웃 고정:

```java
return builder.clone().baseUrl(baseUrl).requestFactory(requestFactory(props)).build();
```

`.clone()`은 Boot가 미리 구성한 빌더(Micrometer observation 포함 → traceparent 자동 전파)를 물려받기 위함. 4개 빈이 같은 타입이라 **생성자 파라미터 이름**(`settlementRestClient` 등)으로 주입 매칭된다(Boot가 `-parameters` 컴파일을 켜둔 덕분 — C++ 같으면 이름이 안 남지만 Java는 켜면 남는다).

### admin context 전파

모든 다운스트림 호출에 헤더를 찍어 각 service가 "누가/어떤 액션/어떤 trace"인지 자기 audit에 남기게 한다:

```java
headers.set("X-Admin-Actor-Id", context.actor().id());
headers.set("X-Admin-Actor-Role", context.actor().roleName());
headers.set("X-Admin-Action-Id", context.actionId().toString());  // UUID v7
```

`AdminContext`는 컨트롤러 메서드 파라미터로 선언만 하면 `HandlerMethodArgumentResolver`가 SecurityContext의 Jwt + 새 actionId + MDC traceId로 만들어 주입한다.

### UUID v7 직접 생성

actionId는 영속 *전에* 필요(헤더로 보내야 하니)해서 Hibernate `@UuidGenerator(style=TIME)`를 못 쓴다. RFC 9562 v7을 직접 생성(48-bit ms 타임스탬프 prefix → 시간순 정렬):

```java
long msb = (timestamp << 16) | VERSION_7_BITS | randA;   // 0x7000 = 버전 7
long lsb = VARIANT_BITS | (random & RAND_B_MASK);         // 0b10 변형
```

### 에러 처리 = thin proxy의 투명성

다운스트림 non-2xx는 **그대로 relay**(모든 service가 이미 problem+json이라 운영자가 진짜 이유를 본다). 도달 불가/타임아웃만 502/504로 변환:

```java
catch (RestClientResponseException e) { throw DownstreamStatusException.from(e); }  // 상태+바디 보존
catch (RestClientException e) { throw new DownstreamUnavailableException(service, e); }  // 502/504
```

`@PreAuthorize` 거부(`AccessDeniedException`)는 advice가 403으로(필터단 핸들러와 동일 모양).
