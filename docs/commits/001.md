# 001 · feat(auth): 운영자 보안 — RS256 JWT, role 가드, IP allowlist

## 개요

admin-api는 보안 critical한 표면이다. 비즈니스 로직보다 **누가(인증) 무엇을(인가) 어디서(IP)** 할 수 있는지가 먼저다. ADR-0011: 운영자 토큰은 RS256 JWT(`role` claim), 발급은 별도 IAM, admin-api는 **검증만**(공개키는 환경변수).

## 작업 순서

1. `AdminRole` enum(ADMIN/TRADER/CS/READONLY) + `role` claim → `ROLE_*` 권한 매핑.
2. `AdminSecurityProperties`(공개키 PEM, role claim 이름, IP CIDR 목록) 바인딩.
3. `RsaKeys` — PEM → `RSAPublicKey` 파싱(키 없으면 startup 실패).
4. `SecurityConfig` — stateless resource server + `JwtDecoder`(RS256) + 필터체인 + 메서드 보안.
5. `IpAllowlistFilter` — `/admin/**`만 CIDR 검사, bearer 필터보다 앞.
6. `ProblemDetailSupport` — 401/403을 RFC 7807로 직접 렌더.
7. `WhoAmIController`(`/admin/v1/whoami`) + 인증/인가 테스트.

## 작업 내역

### RS256 검증기 (JwtDecoder)

핵심. 공개키(PEM)로 서명만 검증한다. C/C++로 치면 OpenSSL로 RSA 서명 verify하는 것과 같지만, Spring은 `NimbusJwtDecoder`가 다 해준다:

```java
RSAPublicKey publicKey = RsaKeys.parsePublicKey(properties.jwt().publicKey());
NimbusJwtDecoder decoder = NimbusJwtDecoder.withPublicKey(publicKey)
    .signatureAlgorithm(SignatureAlgorithm.RS256)   // ★ RS256 고정 (알고리즘 혼동 방어)
    .build();
```

`.signatureAlgorithm(RS256)` 고정이 보안상 중요하다. 이걸 안 하면 `alg=none`이나 HS256(공개키를 HMAC 비밀로) 같은 알고리즘 혼동 공격 여지가 생긴다(004 보안 테스트에서 검증).

### role claim → 권한 매핑

JWT의 단일 `role` claim("admin"/"trader"/...)을 Spring 권한 `ROLE_<NAME>`으로 변환:

```java
converter.setJwtGrantedAuthoritiesConverter(jwt ->
    AdminRole.fromClaim(jwt.getClaimAsString(roleClaim))
        .map(r -> List.of(new SimpleGrantedAuthority(r.authority())))
        .orElseGet(List::of));
```

그러면 컨트롤러에서 `@PreAuthorize("hasAnyRole('ADMIN','TRADER')")`로 endpoint별 가드. ADMIN을 매 가드에 명시(선형 계층이 아니라 TRADER/CS가 서로 다른 액션을 맡기 때문).

### 필터 체인 순서

```
/admin/** → IpAllowlistFilter(CIDR) → BearerTokenAuthenticationFilter(RS256+exp) → @PreAuthorize
```

`IpAllowlistFilter`를 bearer 필터 *앞*에 둬서 allowlist 밖 IP는 인증 전에 차단. health/readiness/prometheus는 `permitAll`(k8s probe·스크레이프), 나머지는 `denyAll` 기본.

### RFC 7807 보안 에러

Spring Security 기본 401/403은 RFC 7807이 아니다. 직접 `AuthenticationEntryPoint`(401)·`AccessDeniedHandler`(403)를 `ProblemDetailSupport`로 렌더. shared-protocol `ErrorCode`엔 UNAUTHORIZED/FORBIDDEN이 없어서(의도적 — 투기적 enum 회피) 타입 URI를 로컬 정의.

## 결과

`AdminSecurityTest` 9개 통과. DB·Kafka 없이(autoconfigure exclude) 실제 보안 필터체인을 부팅해 MockMvc로 검증. 테스트가 in-test로 RS256 토큰을 민팅(Nimbus)해 valid→200, expired/wrong-key/tampered/malformed/missing→401, role 부족→403, IP 밖→403.

## 요약

- RS256 고정 검증기 + `role`→권한 매핑 + endpoint별 `@PreAuthorize`.
- IP allowlist를 인증 앞단에. 전 실패 RFC 7807.
- 공개키 없으면 부팅 거부(인증 사실상 비활성 상태 방지).

## 기억/설명 Level

- **L1**: role 4종, 어떤 endpoint가 어떤 role인지 표.
- **L2**: RFC 7807 보안 에러 렌더, IP allowlist 필터가 `/admin/**`만 거는 이유, health permitAll.
- **L3**: **RS256 검증기 구성 + `alg` 고정의 보안 의미**, role claim→권한 변환, 필터체인 순서(IP→bearer→preauthorize)와 그 이유. (면접 단골)
