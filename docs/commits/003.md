# 003 · feat(audit): 모든 admin 액션을 Kafka + audit_log에 이중 박제

## 개요

ADR-0011의 핵심 책임. 운영 액션은 돈·계정을 건드리므로 "누가 언제 무엇을 했나"가 사고 시 1차 증거다. 두 독립 저장소에 같은 기록을 남겨(action id로 cross-verify) 한 쪽이 의심스러우면 다른 쪽과 대조한다:

- **내구(durable)**: `audit_log` 테이블(admin-api가 소유하는 유일한 테이블).
- **스트리밍**: Kafka `admin.action` 토픽(`AdminActionRecorded`, Avro).

## 작업 순서

1. Avro 스키마 `AdminActionRecorded.avsc`(avro-maven-plugin이 .java 생성).
2. `AuditLogEntity`(JPA) + `AuditLogRepository` + Flyway `V1__audit_log.sql`.
3. `AvroSerializer` + `AuditKafkaConfig`(byte[] producer) + `AdminActionPublisher`.
4. `AuditService` — DB 먼저, 그다음 Kafka(best-effort).
5. `@Audited` 애너테이션 + `AuditAspect`(AOP) — 모든 액션에 자동 적용.
6. 6개 컨트롤러 메서드에 `@Audited` + `GET /admin/v1/audit-logs`(offset page).

## 작업 내역

### annotation 기반 AOP 감사

핵심 설계. 각 컨트롤러 메서드에 `@Audited(action, target SpEL, reason SpEL)`만 달면 aspect가 결과를 보고 기록한다:

```java
@PostMapping("/{betId}/void")
@PreAuthorize("hasAnyRole('ADMIN','TRADER')")
@Audited(value = AdminAction.SETTLEMENT_VOID, target = "#betId")
public ResponseEntity<Void> voidBet(@PathVariable UUID betId, AdminContext context) { ... }
```

aspect는 `@Order(HIGHEST_PRECEDENCE)`로 `@PreAuthorize`보다 **바깥**에서 돈다. 그래서 권한 거부(403)도 감사된다 — "권한 없는 자가 시도했다"는 보안상 중요한 기록. 성공/실패 + HTTP 상태는 반환값(ResponseEntity) 또는 던져진 예외(relay 상태/403/502/504)에서 도출.

### ★ AOP 인자 바인딩 함정 (시간 까먹은 곳)

처음엔 교과서대로 `@Around("@annotation(audited)")` + 파라미터 `Audited audited`로 바인딩했다. 그런데 전 endpoint가 500:

```
IllegalStateException: Required to bind 2 arguments, but only bound 1 (JoinPointMatch was NOT bound)
```

`argNames` 명시도 안 통했다. 결론은 **바인딩을 버리고 advice 안에서 리플렉션으로 읽기**:

```java
@Around("@annotation(com.sportsbook.admin.audit.Audited)")
public Object audit(ProceedingJoinPoint joinPoint) throws Throwable {
  Audited audited = ((MethodSignature) joinPoint.getSignature())
      .getMethod().getAnnotation(Audited.class);   // 바인딩 대신 리플렉션
  ...
}
```

Java/Spring 처음 학습자의 전형적 함정. C/C++엔 "프록시가 메서드를 감싸고 포인트컷 변수가 파라미터에 바인딩된다"는 레이어 자체가 없다. 리플렉션 방식이 더 단순·견고.

### SpEL로 target/reason 추출

`target = "#betId"`, `reason = "#request.reason()"` 같은 SpEL을 메서드 인자에 대해 평가:

```java
EvaluationContext ctx = new MethodBasedEvaluationContext(
    joinPoint.getTarget(), method, joinPoint.getArgs(), PARAM_NAMES);
Object value = PARSER.parseExpression(expression).getValue(ctx);
```

### 이중 박제 + best-effort 트레이드오프

DB를 먼저(가장 내구) 쓰고 Kafka를 발행. **둘 다 실패해도 응답은 안 깨뜨린다** — 다운스트림 액션은 이미 일어났으니, 감사 실패로 운영자에게 "실패"라 잘못 보고하면 안 된다. 실패는 로그+메트릭(`admin.audit.write.failure`)으로:

```java
try { repository.save(entity); }
catch (RuntimeException e) { meters.counter(FAILURE_METRIC, "store","db").increment(); log.error(...); }
try { publisher.publish(actorId, event); }
catch (RuntimeException e) { meters.counter(FAILURE_METRIC, "store","kafka").increment(); log.error(...); }
```

admin-api는 다운스트림과 한 트랜잭션이 아니므로 이론상 "액션됨/감사 빠짐" 갭 존재(회고 §5의 한계). 무손실은 outbox가 필요하나 V1 스코프 밖.

### Avro byte[] 발행

`admin.action`은 `(String key=actorId, byte[] value)`. Schema Registry 없이 .avsc만(ADR-0014). V1엔 consumer가 아직 없다 — DB가 1차 조회원, Kafka는 사본.

### 조회 endpoint

`GET /admin/v1/audit-logs?from=&to=&actor=` — offset pagination(ADR-0004, admin은 cursor 아님). 파라미터화된 JPQL(`:actor IS NULL OR a.actorId = :actor`)로 SQL injection 차단(004에서 검증). 읽기라 자기 자신은 감사 안 함.

## 결과

`AuditIntegrityTest`(실제 PostgreSQL via Testcontainers + embedded Kafka + WireMock) 3개 통과: 성공 액션이 DB·토픽에 **action id로 일치**(cross-verify), 다운스트림 404는 FAILED/404로 기록, 조회 endpoint가 기록을 newest-first로 반환.

## 요약

- `@Audited` + 최외곽 aspect로 권한 거부 포함 전 액션 자동 감사.
- DB(내구)+Kafka(스트리밍) 이중 박제, action id로 cross-verify.
- best-effort(비-트랜잭션) — 응답 보호 우선, 실패는 메트릭.
- offset 조회 + 파라미터화 쿼리.
