# 004 · test(security): 알고리즘 혼동 · SQL injection · IP bypass 시도

## 개요

admin-api는 보안 critical 표면이라 **공격을 증명**하는 게 1차 deliverable이다(부하 아님). 001의 `AdminSecurityTest`가 위조/만료/wrong-key/tampered/missing을 덮었으니, 여기선 명시적으로 짚어야 할 두 공격을 추가한다: **JWT 알고리즘 혼동**과 **SQL injection**.

## 작업 순서

1. `TestKeys`에 공격 토큰 헬퍼 추가 — `noneAlgToken`(alg=none), `hmacWithPublicKeyToken`(HS256 혼동).
2. `SecurityProbeTest`(실제 PostgreSQL via Testcontainers) — 4 공격 케이스.

## 작업 내역

### JWT 알고리즘 혼동 (RS256 고정의 가치 증명)

001에서 `JwtDecoder`를 RS256으로 고정한 이유를 공격으로 입증한다.

**(1) alg=none** — 서명 자체를 떼어낸 토큰:

```java
return new PlainJWT(claims(subject, role)).serialize();  // alg=none, 서명 없음
```

**(2) RS256→HS256 혼동** — 가장 교묘한 공격. 공개키는 누구나 안다. 검증기가 알고리즘을 토큰 헤더에서 그대로 믿으면, 공격자가 **공개키 바이트를 HMAC 비밀로** HS256 서명한 토큰을 위조할 수 있다:

```java
SignedJWT jwt = new SignedJWT(new JWSHeader(JWSAlgorithm.HS256), claims(subject, role));
jwt.sign(new MACSigner(TRUSTED.getPublic().getEncoded()));  // 공개키를 HMAC 비밀로!
```

둘 다 401이어야 한다. RS256 고정 디코더는 헤더의 alg를 믿지 않고 RS256만 받으므로 거부. (naive 구현이었다면 (2)가 통과해 admin 전체가 뚫린다.)

### SQL injection (파라미터화 증명)

유일한 사용자 제어 쿼리 입력은 audit-logs의 `actor` 필터다. 시드 1행(`actorId="u-real-operator"`) 후:

```java
// 정상: 실제 actor → 1행
get("/admin/v1/audit-logs").param("actor", "u-real-operator")  → totalElements 1
// 공격: 주입 문자열은 리터럴 값으로 바인딩 → 0행 (문자열 결합이었다면 전체 노출)
get("/admin/v1/audit-logs").param("actor", "' OR '1'='1")      → totalElements 0
```

JPQL `:actor`가 PreparedStatement 파라미터로 바인딩되므로 주입이 값으로만 취급된다(실행 안 됨). 0행 = 시드 행도 안 샜다 = 안전.

### IP bypass

유효 토큰이어도 allowlist 밖(`203.0.113.7`)이면 403 `IP_NOT_ALLOWED`. IP 필터가 인증 앞단이라 토큰 유효성과 무관하게 차단됨을 재확인.

## 결과

`SecurityProbeTest` 4개 통과(실제 PG). alg=none·HMAC혼동→401, SQLi→0행(무력화), IP밖→403.

## 요약

- RS256 고정의 가치를 알고리즘 혼동 2종으로 입증.
- 파라미터화 JPQL로 SQLi 무력화 증명.
- IP allowlist가 유효 토큰도 IP로 차단.
