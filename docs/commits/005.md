# 005 · test(load): audit read 경량 k6 baseline harness

## 개요

ADR-0011상 admin-api의 부하는 **의도적으로 낮다**(운영자 ~10명, 분당 수십 건). 그래서 stress가 아니라 "읽기 경로가 적당한 동시성에서 정상인지" 확인하는 경량 harness만 둔다. 증명의 무게는 보안·감사 무결성(001·003·004)에 있다.

## 작업 순서

1. `load-test/scenarios/audit-read.js` — k6 스크립트(가장 무거운 운영 읽기 = audit-logs).
2. `load-test/README.md` — 시나리오·임계값·실행법·왜 후순위인지.
3. `load-test/results/BEST.md` — 정직한 상태 기록(미실행) + 진짜 증명 포인터.

## 작업 내역

### k6 시나리오

30 VU로 `GET /admin/v1/audit-logs`를 30초. 가벼운 임계값:

```js
thresholds: {
  http_req_failed: ['rate<0.01'],                 // 에러 < 1%
  http_req_duration: ['p(95)<100', 'p(99)<200'],  // ms
}
```

토큰은 env(`ADMIN_TOKEN`)로 주입(READONLY+). base URL도 env.

### 정직한 BEST.md

dev 세션에선 풀 스택(admin-api+PG+토큰)을 띄우지 않아 **실측을 안 했다**. BEST.md에 이를 명시하고, 대신 `mvn verify`로 재현되는 진짜 증명(보안 매트릭스·감사 무결성)을 가리킨다. 숫자를 지어내지 않는 게 ADR-0016/회고 원칙.

## 결과

harness는 스택이 서면 즉시 실행 가능. 부하 수치는 후순위로 명시, 핵심 proof는 green한 JUnit 스위트.

## 요약

- 경량 k6 read harness(30 VU, p95<100ms) + README + 정직한 BEST.md.
- 부하는 낮은 우선순위 — 보안·감사가 진짜 deliverable.
