# 005 · test(load): audit read 경량 k6 baseline harness

## 개요

ADR-0011상 admin-api의 부하는 **의도적으로 낮다**(운영자 ~10명, 분당 수십 건). 그래서 stress가 아니라 "읽기 경로가 적당한 동시성에서 정상인지" 확인하는 경량 harness만 둔다. 증명의 무게는 보안·감사 무결성(001·003·004)에 있다.

## 작업 순서

1. `load-test/scenarios/audit-read.js` — k6 스크립트(가장 무거운 운영 읽기 = audit-logs).
2. `load-test/README.md` — 시나리오·임계값·실행법·왜 후순위인지.
3. `load-test/results/BEST.md` — 정직한 상태 기록(미실행) + 진짜 증명 포인터.

## 작업 내역

### k6 시나리오

**책임:** 30 VU로 가장 무거운 운영 읽기(`GET /admin/v1/audit-logs`)를 약 30초 건다. 임계값은 가볍게(에러 < 1%, 지연 p95/p99). 토큰·base URL은 env로 주입(READONLY+). 구조(import·env·options·default)는 계약이라 노출하고, 실행기/stages/threshold 값과 요청 본문이 비울 대상이다.

- 스택이 서면: `ADMIN_BASE_URL=… ADMIN_TOKEN=<READONLY+ jwt> k6 run load-test/scenarios/audit-read.js` → 임계값(에러 < 1%, p95 < 100ms) 통과.

- 정적 점검: `k6 inspect load-test/scenarios/audit-read.js` 로 스크립트가 파싱되고 options(실행기·stages·thresholds)가 의도대로 해석되는지(스택 없이 가능).
