# 001 · feat(auth): 운영자 보안 — RS256 JWT, role 가드, IP allowlist

## 개요

admin-api는 보안 critical한 표면이다. 비즈니스 로직보다 **누가(인증) 무엇을(인가) 어디서(IP)** 할 수 있는지가 먼저다. ADR-0011: 운영자 토큰은 RS256 JWT(`role` claim), 발급은 별도 IAM, admin-api는 **검증만**(공개키는 환경변수).

## 작업 순서

1. `AdminRole` enum(ADMIN/TRADER/CS/READONLY) + `role` claim → `ROLE_*` 권한 매핑.
2. `AdminSecurityProperties`(공개키 PEM, role claim 이름, IP CIDR 목록) 바인딩.
3. `RsaKeys` — PEM → `RSAPublicKey` 파싱(키 없으면 startup 실패).
4. `SecurityConfig` — stateless resource server + `JwtDecoder`(RS256) + 필터체인 + 메서드 보안.
5. `IpAllowlistFilter` — `/admin/**`만 CIDR 검사, bearer 필터보다 앞.
6. `ProblemDetailSupport` — 401/403을 RFC 7807로 직접 렌더.
7. `WhoAmIController`(`/admin/v1/whoami`) + 인증/인가 테스트.

## 작업 내역

### RS256 검증기 (JwtDecoder)

**책임:** 공개키만으로 서명을 검증하는 디코더 빈. 토큰 헤더의 `alg`를 신뢰하지 말고 **RS256으로 고정**해야 알고리즘 혼동(`alg=none`/HS256) 공격을 원천 차단한다(그 가치는 004가 공격으로 입증). issuer가 설정되면 추가로 핀.

```java
@Bean
public JwtDecoder jwtDecoder() {
  // TODO: properties의 공개키 PEM을 RSAPublicKey로 파싱(RsaKeys)하고,
  //       서명 알고리즘을 RS256으로 '고정'한 공개키 기반 JWT 디코더를 만든다.
  //       issuer가 비어있지 않으면 issuer까지 검증하는 validator를 추가한다.
}
```

### role claim → 권한 매핑

**책임:** JWT의 단일 `role` claim 문자열을 Spring 권한 `ROLE_<NAME>` 하나로 변환한다. 미인식/누락 role은 throw가 아니라 **권한 0개**로 둬 모든 `@PreAuthorize`를 실패시킨다(인증됨 ≠ 인가됨). 컨트롤러는 `@PreAuthorize("hasAnyRole('ADMIN','TRADER')")`로 endpoint별 가드. ADMIN은 선형 계층이 아니므로 매 가드에 명시.

```java
private JwtAuthenticationConverter roleConverter() {
  // TODO: properties의 role-claim 이름으로 값을 읽어 AdminRole로 파싱한다.
  //       파싱 성공 → ROLE_<NAME> 권한 1개, 실패/누락 → 빈 권한 목록을 주는 converter를 반환.
}
```

### 필터 체인 순서

**책임:** `/admin/**` 요청을 **IP 검사 → bearer(RS256+exp) → `@PreAuthorize`** 순서로 흐르게 구성한다. IP 필터를 bearer **앞**에 둬야 allowlist 밖 IP가 인증 전에 차단된다. health/readiness/prometheus는 토큰 없이 통과(k8s probe·스크레이프), 그 외 명시되지 않은 요청은 기본 거부.

```java
@Bean
public SecurityFilterChain securityFilterChain(
    HttpSecurity http, JwtDecoder jwtDecoder, ProblemDetailSupport problems) throws Exception {
  // TODO: stateless 세션 + CSRF 비활성(세션/쿠키 없음 — 매 호출이 bearer).
  //       authorize: actuator health/prometheus/info = permitAll,
  //                  /admin/** = authenticated, 그 외 anyRequest = denyAll.
  //       oauth2 resource server에 위 jwtDecoder + roleConverter 연결.
  //       IpAllowlistFilter를 BearerTokenAuthenticationFilter '앞'에 추가.
  //       401(entryPoint)·403(accessDeniedHandler)을 아래 RFC 7807 렌더러로 연결.
}
```

### RFC 7807 보안 에러

**책임:** Spring Security 기본 401/403은 RFC 7807이 아니다. `AuthenticationEntryPoint`(401)·`AccessDeniedHandler`(403)를 `ProblemDetailSupport`로 직접 렌더한다. shared-protocol `ErrorCode`엔 UNAUTHORIZED/FORBIDDEN이 **의도적으로 없으므로**(투기적 enum 회피) 타입 URI를 로컬 정의한다.

```java
// TODO: ProblemDetailSupport.write(response, status, typeUri, title, code, detail, request)로
//       401(UNAUTHORIZED)·403(FORBIDDEN) 응답을 RFC 7807 본문으로 렌더하고 위 필터체인에 연결.
//       IpAllowlistFilter의 차단(403)도 같은 모양(IP_NOT_ALLOWED)으로.
```

## 검증

- `mvn -Dtest=AdminSecurityTest test` → 9 케이스 green. Docker 불필요(DB/Kafka는 autoconfigure exclude, 실제 보안 필터체인만 MockMvc로 부팅). 테스트가 in-test로 RS256 토큰을 민팅: valid→200, expired/wrong-key/tampered/malformed/missing→401, role 부족→403, allowlist 밖 IP→403.
- 수동 점검: `GET /admin/v1/whoami`에 유효 토큰 → 본 actor/role이 응답에 나오는지.
- 게이트 감각: 공개키 미설정으로 부팅 시 `RsaKeys`가 startup 실패를 던지는지(인증 사실상 비활성 상태 방지).
