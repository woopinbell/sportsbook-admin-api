# 문제지 사용 가이드

## 목적

`docs/practice/`는 `docs/commits/`(답지)에서 파생한 문제지다. 각 `NNN.md`는 같은 번호 답지의 제목·개요·작업 순서·논리 단위 제목을 그대로 두고, 코드 본문을 `// TODO: <책임>` 골격으로 비운 것이다. 답을 보지 않고 각 단위를 직접 구현하며 admin-api를 처음부터 쌓아 올리는 용도다.

## 사용법

1. `001.md` → `005.md` 번호 순서대로 진행한다.
2. 각 문서의 `## 개요`·`## 작업 순서`로 무엇을 만들지 잡는다.
3. `### 논리 단위`마다 `**책임:**` 한 줄을 읽고, 코드 골격의 `// TODO`를 실제 구현으로 채운다(시그니처·구조·키는 노출돼 있다 — 본문/값/알고리즘이 풀 문제다).
4. 막히면 같은 번호의 답지 `../commits/NNN.md`를 펼친다.
5. 각 문서 끝 `## 검증`의 명령으로 통과 여부를 확인한다.

## 문서 범위 — 무엇이 있고 무엇이 빠졌나

`001`–`005`가 실체 있는 구현 커밋(`feat(auth)`·`feat(api)`·`feat(audit)`·`test(security)`·`test(load)`)에 1:1 대응한다. 답지의 회고 섹션(`결과`·`요약`·`다음 작업`·`기억/설명 Level`·`핵심 확인`·`diff 반영 점검`)은 문제지에서 제외하고, `핵심 확인`은 실행형 `## 검증`으로 옮겼다.

빠진 번호와 사유(답지만 있고 문제지는 없음):

- **`000` chore(project)** — 형제 service의 검증된 pom/checkstyle/logback을 **그대로 따른** 골격이다. 재유도할 설계가 아니라 일관성 복사라 문제지를 만들면 템플릿 빈칸 채우기(공회전)가 된다. 답지 `../commits/000.md`만 둔다.
- **`006` docs(readme)** — 순수 문서 커밋(README 보안/성능 절). 재유도할 실체가 없어 답지만 둔다.

## 구간 지도

| 문서 | 중심 질문 |
| --- | --- |
| `001` | 운영자 토큰을 어떻게 검증하고(RS256 고정), role·IP로 어떻게 가드하는가? |
| `002` | 운영 액션을 어떻게 다운스트림에 위임하고 admin context를 전파하는가? |
| `003` | 모든 액션을 어떻게 자동으로(annotation+AOP) DB·Kafka에 이중 박제하는가? |
| `004` | RS256 고정·파라미터 바인딩의 가치를 어떤 공격으로 증명하는가? |
| `005` | 낮은 운영 부하를 어떤 경량 harness로, 어떻게 정직하게 다루는가? |

## 검증 인프라 주의

- Docker 불필요: `001`(AdminSecurityTest)·`002`(DelegationTest·Uuid7Test)는 그대로 `mvn -Dtest=… test`로 돈다.
- Docker 필요: `003`(AuditIntegrityTest)·`004`(SecurityProbeTest)는 Testcontainers PostgreSQL(+embedded Kafka)을 쓴다 — Docker 데몬이 떠 있어야 한다.
- 풀 스택 필요: `005`(k6)는 admin-api+PG+토큰이 떠야 실측된다(dev 기본 미실행 — 정직성 원칙).
