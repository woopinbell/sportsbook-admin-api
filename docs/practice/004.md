# 004 · test(security): 알고리즘 혼동 · SQL injection · IP bypass 시도

## 개요

admin-api는 보안 critical 표면이라 **공격을 증명**하는 게 1차 deliverable이다(부하 아님). 001의 `AdminSecurityTest`가 위조/만료/wrong-key/tampered/missing을 덮었으니, 여기선 명시적으로 짚어야 할 두 공격을 추가한다: **JWT 알고리즘 혼동**과 **SQL injection**.

## 작업 순서

1. `TestKeys`에 공격 토큰 헬퍼 추가 — `noneAlgToken`(alg=none), `hmacWithPublicKeyToken`(HS256 혼동).
2. `SecurityProbeTest`(실제 PostgreSQL via Testcontainers) — 4 공격 케이스.

## 작업 내역

### JWT 알고리즘 혼동 (RS256 고정의 가치 증명)

**책임:** 001에서 `JwtDecoder`를 RS256으로 고정한 이유를 공격으로 입증한다. (1) **alg=none** — 서명 자체를 떼어낸 토큰. (2) **RS256→HS256 혼동** — 가장 교묘: 공개키는 누구나 안다. 검증기가 헤더의 `alg`를 믿으면 공격자가 **공개키 바이트를 HMAC 비밀로** HS256 서명해 위조할 수 있다. 둘 다 401이어야 한다(RS256 고정 디코더는 헤더 alg를 안 믿고 RS256만 받음 — naive 구현이면 (2)가 통과해 admin 전체가 뚫린다).

```java
// TestKeys — 공격 토큰 헬퍼 (시그니처=계약 노출, 민팅 본문=비움)

  // TODO: 서명 없는 alg=none 토큰을 만들어 직렬화 반환.

}
```

### SQL injection (파라미터화 증명)

**책임:** 유일한 사용자 제어 쿼리 입력은 audit-logs의 `actor` 필터. 시드 1행 후 정상 actor는 1행, 주입 문자열은 **값으로만 바인딩**되어 0행(시드 행도 안 샘). 문자열 결합이었다면 전체가 노출됐을 것 — 0행이 곧 안전의 증거.

```java
@Test
void sqlInjectionInActorFilterIsParameterizedAndMatchesNothing() throws Exception {
  // TODO: actorId 한 행을 시드.
  //       actor=<실제 actorId> → totalElements 1 (sanity).
  //       actor=<주입 문자열, 예: ' OR '1'='1> → totalElements 0 (리터럴로 바인딩 = 무력화).
}
```

### IP bypass

**책임:** 유효 토큰이어도 allowlist 밖이면 403 `IP_NOT_ALLOWED`. IP 필터가 인증 앞단이라 토큰 유효성과 무관하게 차단됨을 재확인한다.

```java
@Test
void requestFromDisallowedIpIsRejectedEvenWithValidToken() throws Exception {
  // TODO: 유효 토큰 + remoteAddr를 allowlist 밖(예: 203.0.113.7)으로 설정 → 403 + errorCode "IP_NOT_ALLOWED".
}
```

## 검증

- `mvn -Dtest=SecurityProbeTest test` → 4 케이스: alg=none·HMAC혼동 → 401, SQLi → 0행(무력화), IP 밖 → 403.
  ⚠ 실제 PostgreSQL(Testcontainers)에 주입을 돌리므로 **Docker 데몬이 떠 있어야** 실행된다. Docker 없는 환경에선 컨테이너 기동 단계에서 실패(설계상 인프라 의존 — 코드 결함 아님).
- 001 `AdminSecurityTest`(Docker 불필요)와 합쳐 README 보안 매트릭스를 완성한다.
